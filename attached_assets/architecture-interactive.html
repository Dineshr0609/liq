Core Architecture Principles

🤖 AI-Native Design
AI is woven into every workflow — from PDF ingestion through 3-stage pipeline extraction to payment calculation. Human-in-the-loop validation ensures accuracy.
🔄 Contract-Agnostic Engine
The Universal Calculation Engine handles any contract type through a dynamic formula evaluator — no code changes needed for new contract types.
🏢 Multi-Tenant Architecture
Multi-location context switching with hierarchical RBAC ensures data isolation across organization, company, and user levels.
🗺️ Knowledge Graph Model
Contracts modeled as knowledge graphs with nodes and edges, enabling rich AI reasoning and cross-contract intelligence.
🏦 Financial Automation
Auto-accrual pipeline: every calculation automatically creates accruals, journal entries (DR 6100 / CR 2200), and feeds period close readiness.
🛡️ Enterprise Ready
Audit trails, version control, approval workflows, branded email templates, lead management, and comprehensive logging.

AI Architecture & Fallback System

Database-driven multi-provider AI system with automatic failover, configurable via System Settings. No restart required for provider changes.

Provider Fallback Chain

⚙️
Configured Provider
From system_settings table
PRIMARY
Reads aiProvider + aiModel from database. Can be Claude, Groq, or OpenAI — configurable via System Settings page. Cached for 60 seconds.
→ fails →
🧠
Anthropic Claude
claude-sonnet-4-5
FALLBACK 1
If the configured provider fails and isn't already Claude, falls back to claude-sonnet-4-5 with 3 retry attempts.
→ fails →
⚡
Groq
llama-3.3-70b-versatile
FALLBACK 2
If Claude also fails, falls back to Groq for ultra-fast inference with rate limit detection and backoff.
🌐 OpenAI (Selectable Primary Only)
OpenAI (gpt-4o, gpt-4o-mini) can be configured as the primary provider via System Settings, but is not in the automatic fallback chain. When set as primary and it fails, the system falls back to Claude → Groq.
⚙️ How the Fallback System Works
▼
Step 1: Read Configuration from Database
System reads aiProvider + aiModel from system_settings table. Default: Anthropic Claude. Settings cached for 60 seconds. No restart required.
Step 2: Try the Configured Primary Provider
Sends request to configured provider — Anthropic Claude, Groq, or OpenAI. Temperature 0.1 for structured JSON output. If successful, returns immediately.
Step 3: Fallback to Claude (if primary failed)
Automatic fallback to claude-sonnet-4-5 — 3 retry attempts with exponential backoff (2^n × 1000ms). Skipped if Claude was already primary.
Step 4: Fallback to Groq (if Claude also failed)
Falls back to Groq (llama-3.3-70b). Explicit rate limit detection with logged wait times. Skipped if Groq was already primary.
Step 5: Safe Empty Response (all providers exhausted)
Each service has internal catch blocks that return valid empty JSON structures — never crashes the pipeline. Error logged and returned to user.

Document Chunking Strategy

Layout-Aware Chunking (layoutAwareChunking.ts)
Strategy
Layout-Aware — preserves document structure (tables, headers, lists)
Chunk Size
3,000 characters (max per chunk)
Chunk Overlap
200 characters — ensures context continuity between chunks
Table Handling
Detects Markdown & ASCII tables, keeps them intact within a single chunk
Header Detection
Recognizes "SECTION X", "ARTICLE II", Markdown headers as section boundaries
List Preservation
Keeps numbered and bulleted lists together
Page Detection
Detects "Page X of Y" patterns, associates page numbers with chunk offsets
Importance Score
0–1 score based on payment keywords ("royalty", "fee", "rate", "threshold", "tier")
Why Layout-Aware?
  • Naive character splitting breaks tables mid-row, making them useless for AI interpretation
  • Section-aware boundaries keep related clauses together for better retrieval relevance
  • Importance scoring allows prioritization of payment-related chunks during retrieval

Hybrid Search & Intent Detection

Before vector search, the RAG pipeline detects the user's intent and routes accordingly:
Data Query Detection
If question is "List my active contracts" or "How many calculations?" → executes direct SQL query instead of vector search. Faster, exact results.
Platform vs Contract Intent
Compares similarity scores between contract embeddings and system embeddings to auto-detect whether question is about platform features or specific contract content.
Fallback to Keyword Search
If HuggingFace embedding service is unavailable (timeout), falls back to text-based keyword search for system docs or uses contract analysis summaries.

Multi-Layer Prompt Architecture

The platform uses a sophisticated 4-layer prompt system for all AI operations:
Layer 1: System Prompt
Defines AI persona as "expert contract analyst". Enforces strict JSON output format. Sets accuracy rules and behavioral constraints.
Layer 2: Instruction Layer
Operation-specific requirements: "Extract EVERY payment rule", "MANDATORY: Include sourceSpan with exact quotes from contract text", structured field requirements.
Layer 3: Contract-Type Layer
Custom prompts fetched from database per contract type code (direct_sales, distributor, royalty_license). Stored in defaultContractTypePrompts.ts.
Layer 4: Few-Shot Examples
Detailed sample JSON outputs included in prompts to guide AI structure. Complete example objects with all expected fields and realistic values.

Prompt Templates by Operation

Extraction Entity & Rule Extraction
Zero-shot prompts for extracting parties, dates, territories, payment rules, tier structures, conditions, and formulas. Mandatory sourceSpan with exact contract text quotes. Outputs structured JSON arrays.
3-Stage Pipeline Clause → Rule → Conflict
Stage A: "contract clause segmentation expert" prompt. Stage B: "rule template mapping" with T1–T12 templates + two-pass validation. Stage C: "conflict detection" analyzing overlapping rules.
Enrichment Deep Rule Analysis
Adds calculation logic, category tags, implementation checklists, real-world examples, and source citations. Processes in batches of 5 to manage rate limits. Auto-triggered on first "Manage Rules" click.
RAG Mode Chunk Processing
Specialized prompts for processing document "chunks" with mandatory citations. Each extracted fact must link to its source chunk. Designed for layout-aware chunks with preserved table context.
Q&A liQ AI Answers
RAG prompt with retrieved chunks as context. Strict accuracy rules: "use ONLY provided info, never speculate". Identity rules prevent revealing underlying AI technology.
Validation Ensemble Checking
Multi-model validation where different providers cross-check each other's extractions. Hallucination detection by comparing extracted values against source text.

JSON Sanitization & Repair Pipeline

🔧 Robust JSON Recovery
AI models frequently return malformed JSON. The platform implements a multi-stage repair pipeline:
sanitizeAIPlaceholders()
Converts AI placeholder values: [NA] → [], "NA" → null, "N/A" → null
extractAndRepairJSON()
Multi-step: strip markdown fences → find first valid JSON block → attempt parse → if truncated, close missing braces/brackets
Truncation Recovery
Detects incomplete JSON (hit max tokens) and attempts to close open arrays/objects to salvage partial results
Safe Fallback
If all repair attempts fail, returns empty but valid JSON structure (empty arrays/objects) — never crashes the pipeline

3-Stage Contract Processing Pipeline

AI-driven extraction pipeline that transforms raw contract text into structured calculation rules with conflict detection and two-pass validation.

Pipeline Overview

Stage A
Clause Segmentation
Break contract into categorized clauses with unique IDs (CL-01, CL-02...)
→
Stage B
Rule Template Mapping
Map financial clauses to 12 standardized rule templates (T1–T12)
→
Stage C
Conflict Detection
Identify overlapping rules, contradictions, and precedence issues

Stage A: Clause Segmentation

Breaking contracts into logical units
Section Identification
AI identifies and separates contract sections using structural cues (SECTION, ARTICLE, numbered clauses).
Clause Extraction
Each clause receives a unique identifier (e.g., CL-01), full text, and section reference.
Categorization
Clauses categorized as: financial calculation, operational, adjustment, general terms, or definitions.
Accrual Impact Flag
Each clause tagged with "affects accrual" flag indicating financial impact for downstream Finance Hub processing.

Stage B: Rule Template Mapping

12 Standardized Rule Templates (T1–T12)
Financial clauses from Stage A are mapped to specific calculation templates:
T1
Percentage of Revenue
T2
Fixed Fee / Flat Rate
T3
Tiered Rate Structure
T4
Volume-Based
T5

T6
Maximum Cap
T7
Sliding Scale
T8

T9
Display / Placement
T10
Rebate / Discount
T11
Revenue Share
T12
Custom / Hybrid
Two-Pass Validation
Pass 1: Heuristic Validation
Checks for missing rates, source text mismatches, and extraction confidence scoring.
Pass 2: AI Self-Validation
AI re-verifies extracted parameters against source text to minimize hallucinations. Cross-references rates, tiers, products, and territories.
Parameters extracted per rule: rates, volume tiers, base metrics (net sales), eligible products, territories, and calculation frequencies.

Stage C: Conflict Detection

Identifying contradictions and overlaps
Analyzes the complete rule set for logical issues:
Overlapping Products/Territories
Same product + territory combination with different rates across multiple rules.
Contradictory Conditions
Mutually exclusive conditions applied to the same transaction scope.
Precedence Ambiguity
Multiple rules could apply to the same transaction with no clear priority order.
Each conflict receives an identifier (e.g., CONF-01), a detailed reason, and an AI-suggested resolution. Human review queue allows approve/reject/override.

Per-Table Rule Extraction

When a contract PDF contains multiple pricing tables, the AI processes each table separately:
1. Layout-Aware Chunking
Document split preserving table boundaries and headers (3000 char chunks, 200 char overlap)
2. Table Detection
Each pricing/rate table identified and isolated (Markdown + ASCII table detection)
3. Individual Table Extraction
AI processes each table with its surrounding context via contract-type-aware prompts
4. Hallucination Detection
Cross-validates extracted values against source text (ensembleValidationService)
5. Rule Synthesis & Dedup
Merges results, removes duplicates, assigns extraction order

When a contract fee calculation completes via the Calculation Engine, the system automatically creates: (1) accrual record linked to the contract, (2) audit trail entry, (3) calculation trace with step-by-step breakdown, (4) draft journal entry with DR (Contract Fee Expense 6100) / CR (Accrued Contract Fees Payable 2200) line items.

Finance Hub Pages

🏦 Accrual Management
AI insight cards, accrual table with confidence scores, detail panel. Live CRUD: approve/reject accruals. Linked to calculation source.
📄 Journal Entry Hub
Stage pipeline: Draft → Pending Review → Approved → Posted. DR/CR line items. ERP sync status. Bulk approve capability.
📅 Period Close Workspace
Close readiness score. Automated checklists. Blocker cards with resolution. Contract close status table. "Auto-populate" maps accruals/JEs into close records.

Period Close Master Data

System Settings — Finance Hub Tab (10th Tab)
Manage period definitions (create/edit/delete fiscal periods), period close checklists (define required items per period), and view audit trail. "Auto-populate" button maps existing accruals and JEs into contract close status records and recalculates readiness scores.

Security Implementation

🔐 Session Management
Passport.js with PostgreSQL session store. Secure cookies, session expiry, concurrent control.
🏢 Multi-Location Filtering
Context switcher in navigation. All queries filtered by active organization context.
📝 Audit Trail
Every action logged with user, timestamp, entity, and change details.
🔒 API Authorization
Middleware checks authentication and role permissions on every endpoint.
🛡️ Input Validation
Zod schemas validate all requests. Drizzle ORM prevents SQL injection.
🔑 Secret Management
API keys as environment secrets. SMTP passwords masked in API responses (smtpPasswordSet boolean).
