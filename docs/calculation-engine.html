1. Overview

The engine turns three things — a contract, its rules, and a set of sales — into one signed number per period: the payable. Two execution paths exist:

  • dynamicRulesEngine.ts — legacy per-sale loop. Still drives the bulk of contract types (royalty, distribution, rebate).
  • universalCalculationEngine.ts — newer phase-ordered engine with full traceability per (rule × bucket × line). Both engines share ruleMatchingUtils.ts for sale-to-rule matching.

Period-close evaluators (periodCloseEvaluators.ts) layer on top of either path to run the netting, flooring, capping, and penalty math. Obligation accruals run independently in obligationsService.ts.

3. Phase ordering

Rules carry a calcPhase (auto-derived from ruleType in shared/calcPhases.ts). The dispatcher executes phases in this fixed order, secondary-sorted by rule priority:

OrderPhaseWhat it does
1gross_calcCompute the raw fee from sales lines.
2returns_offsetSubtract returns, chargebacks. Mutates running total.
3net_adjustmentApply deductions, recoupments, overpayment offsets.
4floor_minimumIf running total < minimum guarantee, bump to MG.
5cap_maximumIf running total > cap, clamp to cap.
6penaltyAdd late-payment / missed-milestone / compliance penalties.
—obligation_accrualSide-channel: book balance-sheet liabilities. Does not move the payable.
Verify-then-pay. A rule must have approvalStatus = 'approved' before the dispatcher will let it run. A rule under "AI extracted" or "under review" is silently skipped. This is the gate behind the traffic-light icons on the rule list.

5. Gross calc rules — math

percentage

implemented
Per-sale · Inputs: baseRate, baseMetric
fee = saleAmount(baseMetric) × baseRate

Simplest path. baseMetric picks which sale field is multiplied (net_sales, gross_sales, units, invoice_amount, custom).

tiered (per-sale ladder)

implemented
Per-sale · Inputs: volumeTiers, tierBasis (units|amount), tierMode (whole|marginal)
for each sale:
  bracket = highest tier whose min ≤ sale.value
  fee     = sale.value × bracket.rate
No aggregation. Each line is independently bracketed — a single $20K sale may land in tier 1 even if the period total is $1M. For period-aggregated tiering use rebate_tiered.

rebate_tiered (retroactive — aggregate, then tier)

implemented
Aggregate · Inputs: volumeTiers, tierBasis, tierMode, formulaDefinition.aggregationPeriod (annual|quarter|month)
1. bucket sales by aggregationPeriod (quarter is default)
2. periodTotal = sum(sale.value) per bucket
3. matchedTier = lookup(periodTotal, tiers, tierMode)
4. periodFee   = periodTotal × matchedTier.rate    // whole mode
              OR Σ band-by-band marginal math       // marginal mode
5. allocate periodFee back to each sale proportionally
   so the audit trail shows "this line contributed $X.YZ"

The aggregationPeriod knob (added April 2026) lets annual-measurement contracts accumulate the full year before tier lookup, instead of always using quarters.

milestone_tiered (cumulative threshold)

implemented
Aggregate, cumulative-to-date · Inputs: volumeTiers, tierBasis
cumulative = sum(sale.value) for all periods up to and including current
matchedTier = lookup(cumulative, tiers)
ratchet: once a tier is reached, the rate stays at that level
         (or applies retroactively if formulaDefinition.retroactive=true)

net_sales_tiered ($-banded ladder)

partial
Per-sale · Currently shares the tiered evaluator

Treated as tiered with tierBasis = amount implied. AI extraction template for "Annual Net Sales" tables targets this type.

promotional_rebate (campaign-windowed)

implemented
Aggregate · Inputs: formulaDefinition.activeFrom, activeTo, plus standard tiered fields
filteredSales = sales where activeFrom ≤ sale.date ≤ activeTo
then: same flow as rebate_tiered against filteredSales

bonus_rebate (single-step kicker)

implemented
Aggregate · Inputs: formulaDefinition.threshold, bonusAmount or bonusRate
periodTotal = sum(sale.value) per period
fee = periodTotal ≥ threshold ? bonusAmount (or periodTotal × bonusRate) : 0

fixed_fee · per_unit · annual_fee · milestone_payment

implemented
Event / one-shot rules
fixed_fee:        fee = formulaDefinition.amount  (per period or per contract)
per_unit:         fee = quantity × ratePerUnit
annual_fee:       fee = formulaDefinition.amount  (booked once per contract year)
milestone_payment: fee = formulaDefinition.amount on milestone trigger

rate_structure / formula_based

partial
Custom math · Inputs: formulaDefinition.rates[] or formulaDefinition.expression
rate_structure:   pick first matching {when, rate} pair, then percentage math
formula_based:    universalFormulaEvaluator runs an arithmetic AST
                  (limited to + - * / parens, sale field refs, constants)

6. Returns offset rules

returns_offset

implemented
returnedAmount = Σ sale.amount where transactionType = 'return'
runningTotal -= returnedAmount × (formulaDefinition.rate ?? grossRuleRate)

chargeback

implemented

Same evaluator as returns_offset, but the resulting line carries reason chargeback for ledger reporting.

7. Net adjustment rules

deduction · discount

implemented
fixed amount:   runningTotal -= formulaDefinition.amount
percentage:     runningTotal -= runningTotal × formulaDefinition.rate

overpayment_offset · advance_recoupment

implemented
availableBalance = open obligation balance for kind
drained          = min(runningTotal, availableBalance)
runningTotal     -= drained
obligation.balance -= drained   // mutate obligation row

8. Floor / minimum rules

minimum_guarantee · annual_minimum · quarterly_minimum · mgr

implemented
Inputs: minimumGuarantee, plus implicit scope from rule type
scope        = annual | quarter | contract  (derived from ruleType)
periodTotal  = sum runningTotal in scope
if periodTotal < minimumGuarantee:
   shortfall = minimumGuarantee - periodTotal
   runningTotal += shortfall
   audit:     "MG top-up"

9. Cap / maximum rules

cap · period_cap · contract_cap

implemented
scope    = period | contract  (derived from ruleType)
ceiling  = formulaDefinition.cap (or rule.maximumCap)
if runningTotal > ceiling:
   excess        = runningTotal - ceiling
   runningTotal -= excess
   audit:         "Cap clamp"

11. Penalty rules

late_payment_penalty

implemented
daysOverdue = today - dueDate
penalty     = outstandingPayable × dailyRate × daysOverdue

missed_milestone_fee

implemented
for each obligation o where o.kind = 'milestone_payment'
                            AND o.status NOT IN ('completed','cancelled')
                            AND o.plannedReleaseDate < today:
  penalty += formulaDefinition.amount

12. Aggregation: tier basis · tier mode · period

Tier basis

Controls which sale field the tier ladder is matched against:

  • units — match the quantity total against tier minimums.
  • amount — match the net_amount total.
  • auto (legacy default) — heuristic: if max(tier.min) < 500,000 assume units, else amount.

Set explicitly via the rule's tier_basis column or formulaDefinition.tierBasis.

Tier mode

  • whole (default) — once a tier is matched, the entire periodTotal is multiplied by that tier's rate.
  • marginal — walk the tiers low→high; each band's eligible amount is multiplied by its own rate. The displayed "rate" becomes a blended weighted average.
// whole mode example: $749,832 vs tiers [$0=0, $175K=2%, $450K=2.5%, $600K=3%]
matchedTier = $600K bracket (3%)
fee         = 749,832 × 3% = $22,494.97

// marginal mode example, same data:
band [0       - 174,999]: 175,000 × 0%  = 0
band [175,000 - 449,999]: 275,000 × 2%  = 5,500
band [450,000 - 599,999]: 150,000 × 2.5%= 3,750
band [600,000 - ∞      ]: 149,832 × 3%  = 4,494.96
fee = 13,744.96   (blended ~1.83%)

Aggregation period new

Controls the bucketing for aggregate-tiered rules (rebate_tiered, milestone_tiered, promotional_rebate, bonus_rebate):

  • annual — one bucket per year. Use for "annual measurement period" contracts.
  • quarter (default) — one bucket per fiscal quarter.
  • month — one bucket per calendar month.

Set via formulaDefinition.aggregationPeriod. Without this knob the engine always bucketed quarterly, which would cause an annual rebate contract to under-pay (each quarter independently below the floor).

13. Sale → rule matching (scope)

Before a rule sees a sale, ruleMatchingUtils.ts filters the sale set down to what this rule should "see":

  1. Contract match — sale's matched_contract_id must equal this rule's contract.
  2. Qualifier match — the rule's qualifiers (Object · Field · Operator · Value) are evaluated. in = inclusive; not_in = exclusive (catch-all rules).
  3. Territory match — wildcards (*, all, empty, "primary territory") match anything; otherwise substring or exact match.
  4. Channel / SKU match — same wildcard semantics.
  5. Best match — when multiple rules qualify a sale, the one with the most specific qualifiers wins (specificity = number of non-wildcard qualifiers).
