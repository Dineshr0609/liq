9. Where Does Base Metric Come From?

The Base Metric dropdown (Net Sales, Gross Sales, Units, etc.) you see on the Rule Edit form tells the system which dollar amount or quantity to use when calculating the fee. Here is exactly where those values come from and how they map to real database fields:

Current Source: Hardcoded in Frontend

The dropdown values are currently defined as a constant array in the frontend code. A database table exists with the same values, but the dropdown does not yet fetch from it.

pipeline-results.tsx → Hardcoded BASE_METRICS array with 8 options
base_metrics (DB table) → Same 8 values seeded into the database (exists but not used by dropdown yet)
GET /api/pipeline/base-metrics → API endpoint exists, returns all rows from base_metrics table
Future improvement: The dropdown should fetch from the API /api/pipeline/base-metrics instead of using the hardcoded array, so new metrics can be added via the database without code changes.

How Base Metric Maps to Sales Data Fields:

When the calculation engine runs, the Base Metric tells it which column from the sales_data table to use as the dollar amount for the formula. Here is the exact mapping:

Base Metric (Dropdown)Sales Data Column UsedWhat It Means
Net Salessales_data.net_amountSale amount after returns and deductions
Gross Salessales_data.gross_amountFull sale amount before any deductions
Unitssales_data.quantityNumber of units sold (not dollars)
Marginsales_data.gross_amount - costProfit margin (revenue minus cost)
Invoice Amountsales_data.gross_amountTotal billed amount (similar to gross)
Subscription Revenuesales_data.net_amountRecurring subscription amounts
Outstanding Balancecustom_fieldsAmount currently owed
Othercustom_fieldsCustom metric defined per contract

Where the Mapping Happens in Code:

// In calculationService.ts (lines 507-540)
// The system checks what base_metric was set on the rule:

if (basis includes "net")      → use sale.netAmount
if (basis includes "gross")    → use sale.grossAmount
if (basis includes "unit")     → use sale.quantity
otherwise                      → default to sale.grossAmount

In Plain Language

When you pick "Net Sales" from the Base Metric dropdown, you're telling the system: "When calculating the fee, use the net sale amount (after deductions) as the number you multiply the rate against."

If you pick "Units" instead, the system uses the quantity of items sold, not the dollar amount.

Think of it like choosing which column on a spreadsheet to run your formula against.

In Technical Language

The base_metric field on royalty_rules is persisted as a varchar. During calculation, calculationService.ts reads this value and resolves it to a SalesContext field reference (netAmount, grossAmount, or quantity).

For universal evaluator mode, the metric is encoded into the formula_definition JSON as a ReferenceNode (e.g., {"type": "reference", "field": "netAmount"}). The FormulaInterpreter then resolves this reference against the sales context at runtime.

10. Full Calculation Process After Rule Saving

Using the TechSound "Standard Royalty on Net Sales" example from the screenshot (T1: Percentage Revenue, Base Metric: Net Sales, 5% rate), here is the complete process that happens from the moment you click "Save Rule" to the final fee number.

In Plain Language (What Happens)

  1. You save the rule. The system stores your choices — the rule name, the rate (5%), what to calculate against (Net Sales), which products it applies to, and the priority. It's like writing down instructions: "Take 5% of net sales for these products."
  2. Sales data comes in. Someone uploads a spreadsheet of sales — which products were sold, how many units, what dates, and for how much money.
  3. You hit "Calculate." The system pulls up the contract, finds all the rules, and finds all the sales that belong to this contract.
  4. Matching starts. For each sale, the system asks: "Does this sale match this rule?" It checks: Is this product in the rule's product list? Is this territory covered? Is this within the rule's date range?
  5. The math runs. For each matching sale, the system looks at the Base Metric (Net Sales = use the net_amount column), multiplies it by the rate (5%), and that gives the fee for that sale.
  6. Special handling for quarterly rebates. If the rule is a tiered rebate (like TechSound), the system first groups all sales by quarter, totals up the units, finds the right tier, and THEN applies that tier's rate to each individual sale's dollar amount.
  7. Results are saved. The total fee, plus a detailed breakdown showing each product and its calculated fee, gets stored so you can review, approve, or export it.

In Technical Language (What Happens)

  1. Rule persistence. POST /api/contracts/:id/rules writes to royalty_rules table with rule_type, base_rate, base_metric, volume_tiers, formula_definition (if universal mode), product_categories, territories.
  2. Sales ingestion. CSV upload via SalesDataParser normalizes column names, parses amounts, and inserts into sales_data table with matched_contract_id set via AI semantic matching or manual assignment.
  3. Calculation trigger. POST /api/contracts/:id/calculate calls calculationService.calculateLicenseFees(). Loads all active rules via royalty_rules WHERE contract_id = :id AND is_active = true.
  4. Engine routing. calculationService inspects rule types. If it detects tiered/rebate rules with quarterly keywords and maxTierMin < 500K, it routes to dynamicRulesEngine.calculateRoyalty() (legacy mode). Otherwise, uses the universal FormulaInterpreter.
  5. Legacy mode execution. dynamicRulesEngine groups sales by quarter (getQuarter(sale.transactionDate)), sums quantity per quarter, matches to volume_tiers to find the applicable rate, then calculates: sale.grossAmount * matchedTier.rate for each sale.
  6. Universal mode execution. FormulaInterpreter walks the formula_definition expression tree. A MultiplyNode with child ReferenceNode("netAmount") resolves sale.netAmount from the SalesContext, multiplies by the LiteralNode(0.05).
  7. Result persistence. Inserts into contract_royalty_calculations (total_royalty, breakdown JSONB) and optionally calculation_line_items for granular audit trail.

Visual Step-by-Step Flow:

1
Rule Saved to Database

Your rule choices are written to the royalty_rules table. The Base Metric you selected (e.g., "Net Sales") is stored in the base_metric column. The rate, tiers, products, and territories are all stored alongside it.

INSERT INTO royalty_rules (contract_id, rule_name, rule_type, base_rate, base_metric, 
  template_code, execution_group, product_categories, territories)
VALUES ('303d2cc0-...', 'Standard Royalty on Net Sales', 'percentage', 
  0.05, 'net_sales', 'T1', 'periodic',

2
Sales Data Uploaded & Matched

Sales CSV is uploaded. Each row is stored in sales_data with product name, quantity, gross and net amounts, and date. The system either auto-matches rows to the contract or you assign them manually.

3
Calculation Engine Decides Which Mode to Use

The engine checks all rules for the contract. If it finds tiered rebate rules with quarterly keywords, it uses the legacy engine (groups by quarter first). Otherwise, it uses the universal evaluator (processes each sale individually).

// Decision logic in calculationService.ts:
Has tiered/rebate rules with quarterly keywords? → Legacy Engine (dynamicRulesEngine.ts)
Has formula_definition on rules?                 → Universal Evaluator (formulaInterpreter.ts)
Default fallback                                 → Universal Evaluator
4
Sales Matched to Rules

For each sale, the engine checks if the product name appears in the rule's product_categories array and the territory is in the rule's territories array. Only matching sales proceed to calculation.

CheckSale FieldRule FieldMatch Logic
Productsales_data.product_nameroyalty_rules.product_categoriesFuzzy text match (contains, stripped codes)
Territorysales_data.territoryroyalty_rules.territoriesExact match or empty = match all
Datesales_data.transaction_dateroyalty_rules.effective_date / expiry_dateMust fall within range
Segmentsales_data.custom_fieldsroyalty_rules.customer_segmentsIf specified, must match
5
Base Metric Resolves to a Dollar Amount

The base_metric value determines which column from the sales data is used as the calculation input:

base_metric = "net_sales"    →  use sale.net_amount   = $76,500
base_metric = "gross_sales"  →  use sale.gross_amount = $76,500
base_metric = "units"        →  use sale.quantity     = 300
6
Fee Calculated Per Sale

The formula is applied. For a simple 5% percentage rule on net sales:

// Simple percentage:
fee = sale.net_amount × base_rate
fee = $76,500 × 0.05 = $3,825.00

// Tiered (quarterly aggregation first):
Q1 total units = 1,080 → matches Tier 2 (1,001-5,000) → rate = 5%
fee = sale.gross_amount × tier_rate
fee = $76,500 × 0.05 = $3,825.00
7
Results Stored & Displayed

All individual fees are summed up. The total and the per-product breakdown are saved to contract_royalty_calculations. You see the result on the Calculation Engine page with charts, breakdowns, and the option to approve or reject.

INSERT INTO contract_royalty_calculations
  (contract_id, total_royalty, total_sales_amount, sales_count, breakdown)
VALUES ('303d2cc0-...', 31017.50, 620350.00, 9, '[{...per-product detail...}]')

The Short Version (Plain Language)

Save Rule = Write down the instructions (5% of net sales for these products)

Upload Sales = Give the system the actual sales data to work with

Calculate = The system matches each sale to the right rule, looks up which dollar amount to use (net vs gross vs units), applies the rate, and adds it all up

Result = You get a total fee ($31,017.50) with a line-by-line breakdown showing exactly how each product contributed

The Short Version (Technical)

Persistence: Rule → royalty_rules (base_metric, formula_definition, volume_tiers)

Ingestion: CSV → sales_data (matched_contract_id via semantic matching)

Routing: calculationService inspects rule types → legacy (quarterly agg) or universal (formula tree)

Resolution: base_metric → SalesContext field reference → actual numeric value from sales row

Execution: rate × resolved_amount per sale, with optional quarterly grouping for tiered rules

Storage: contract_royalty_calculations + calculation_line_items with full audit trail
