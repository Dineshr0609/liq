base_rate Storage Convention

The base_rate column stores percentage values as human-readable numbers: 3.50 means 3.50%, NOT 350%. The calculation engine always divides by 100 for percentage-type rules before multiplying against the sale amount. For fixed_fee/per_unit rules, base_rate is used as-is (dollar amount per unit).

  • Percentage rule: gross_amount × (3.50 / 100) = gross_amount × 0.035
  • Fixed fee rule: quantity × 0.25 = $0.25 per unit

Company-Wide Fallback Removed

The previous fallback strategy (which loaded ALL sales for a company when no matched_contract_id records existed) has been permanently removed to prevent cross-contract contamination. When multiple contracts share the same company_id, the fallback would cause every contract to calculate against every sale, producing wildly incorrect results.

Sales must be linked to contracts via one of these methods:

  • Upload with contract selected: When uploading sales data with a specific contract chosen, matched_contract_id is set automatically
  • Smart Match (automatic): When uploading company-wide, the SalesContractMatcher service runs automatically — it writes ALL matches to the sale_contract_matches junction table and marks the best as primary in matched_contract_id
  • Smart Match (manual): Users can click "Smart Match" on the Data Ingestion or Calculate tabs to re-run matching for existing unmatched sales

Smart Sales-to-Contract Matching

server/services/salesContractMatcher.ts — SalesContractMatcher.matchSalesToContracts()

When sales are uploaded company-wide (without selecting a specific contract), the Smart Matcher automatically assigns each sale to the correct contract by analyzing which contract's rules would match the sale.

How Smart Match Works — Plain Language Overview

The Big Picture

When you upload a sales CSV, the sales rows don't automatically belong to any contract. The Smart Match step looks at each sale and figures out which contracts it matches by comparing the sale's product, territory, and channel against the rules in each active contract. A single sale can match multiple contracts — all matches are recorded in the sale_contract_matches junction table, and the best match is also written to the legacy matched_contract_id column for backward compatibility.

Step-by-Step Process

#StepWhat Happens
1 Load All Active Contracts The system pulls every active contract belonging to the same company as the uploaded sales data.
2 Load Rules for Each Contract For each contract, it loads all the contract rules — these contain the product categories, territories, and channels that the contract covers.
3 For Each Sale, Score Every Contract The system takes each sale row and asks: "Does this sale fit any rule in Contract A? Contract B? Contract C?" It checks three dimensions:
  • Product match — Does the sale's product name match any rule's product categories? (e.g., sale for "Russet Potatoes 50lb" matches a rule with category "Potatoes")
  • Territory match — Does the sale's territory match the rule's territories?
  • Channel match — Does the sale's channel match the rule's channel?
4 Specificity Scoring If a sale matches rules in multiple contracts, the system picks the most specific match:
  • A rule for "Potatoes" specifically scores higher than a "General" catch-all rule
  • An exact product name match scores higher than a fuzzy/partial match
  • Ranking: exact match > contains match > category match > general fallback
5 Record ALL Matches + Mark Primary Every contract+rule match is recorded in the sale_contract_matches junction table. The highest-scoring match is marked is_primary = true and its contract ID is also written to the legacy matched_contract_id column for backward compatibility. A single sale matching 3 contracts creates 3 junction rows.
6 Unmatched Sales Sales that don't match any contract rule stay unmatched (no junction rows, no matched_contract_id). These show up in the Data Ingestion page as "unmatched" for manual review.

Simple Example

Say you have two contracts:

  • Contract A: Covers Potatoes and Onions
  • Contract B: Covers Fruits and General Produce

You upload a sale for "Russet Potatoes 50lb":

Technical Deep Dive

Everything Is Dynamic — Nothing Is Hardcoded

The matching engine has zero hardcoded values. It does not know about "Potatoes", "Onions", "SoundPro", or any specific product in advance. All matching criteria come from the database at runtime:

  • Qualifier values are loaded from the contract_qualifiers table — whatever values are stored there (e.g., "Potatoes", "Widget X", "Premium Tier") are used as-is for matching
  • Product categories are loaded from the contract_rules.product_categories[] column — whatever categories the AI extracted (or the user manually entered) are used
  • Contract dates are loaded from the contracts.effective_start / effective_end columns
  • Exclusion prefixes (!) and operator types (in, not_in) are the only "known" patterns — the actual values they apply to are all dynamic

This means: if you add a new contract with rules for "Widgets" and qualifiers for "Premium Widgets", the matcher will automatically match "Premium Widget X" sales to that contract — no code changes needed. The algorithm is a generic matching engine that works with whatever data is in the database.

Where Do Qualifiers and Categories Come From?

Example: Adding a New Contract Tomorrow

If you upload a contract for "Acme Widgets" with rules for "Standard Widgets" and "Premium Widgets", the AI will extract:

The Smart Matcher will then automatically match "Premium Widget Deluxe 2026" sales to Rule 1 (because "premium widget deluxe 2026".includes("premium widgets") is true) and all other Acme sales to Rule 2's General catch-all. No code changes required.

Complete Step-by-Step Algorithm (Pseudocode)

The entire matching process has 3 phases: Data Loading, Sale-by-Sale Matching Loop, and Batch Database Update. Below is the full logic as readable pseudocode.

Phase 1: Load All Data Into Memory

Before any matching begins, the system loads three datasets from the database into memory for the given companyId:

// ─── PHASE 1: DATA LOADING ───────────────────────────────────────

INPUT: companyId (the company whose sales we want to match)

// STEP 1.1: Load all UNMATCHED sales for this company
//   → Only sales where matched_contract_id IS NULL
//   → These are the sales that need a contract assignment
unmatchedSales = SELECT id, product_name, product_code, category, territory, transaction_date
                 FROM sales_data
                 WHERE company_id = companyId
                   AND matched_contract_id IS NULL

// Example result: 613 rows
// Row 1: { id: "sale-001", product_name: "Russet Potatoes 50lb", category: "Potatoes", ... }
// Row 2: { id: "sale-002", product_name: "SoundPro Wireless Headphones - Black", category: null, ... }
// Row 3: { id: "sale-003", product_name: "Yellow Onions 25lb", category: "Onions", ... }

// STEP 1.2: Load ALL contract rules for this company
//   → Joins contract_rules with contracts
//   → Only active contracts (status: analyzed, active)
//   → Rules from ALL review statuses (verified, under_review, ai_extracted)
//   → Ordered by contract_number, then priority
allRules = SELECT cr.id, cr.rule_name, cr.product_categories, cr.territories,
                  cr.contract_id, c.contract_number, c.effective_start, c.effective_end
           FROM contract_rules cr
           JOIN contracts c ON cr.contract_id = c.id
           WHERE c.company_id = companyId
             AND c.status IN ('analyzed', 'active')
           ORDER BY c.contract_number, cr.priority

// Example result:
// Rule A: { contractId: CNT-011, ruleName: "Potatoes Per-Unit Rebate", product_categories: ["Potatoes","Onions"] }
// Rule B: { contractId: CNT-011, ruleName: "General Purchases Rebate", product_categories: ["General"] }
// Rule C: { contractId: CNT-016, ruleName: "Volume Rebate", product_categories: ["SoundPro Wireless...","SoundBar Mini"] }

// STEP 1.3: Load ALL qualifiers for this company's contracts
allQualifiers = SELECT cq.term_id AS rule_id, cq.qualifier_type, cq.operator, cq.qualifier_value
                FROM contract_qualifiers cq
                JOIN contracts c ON cq.contract_id = c.id
                WHERE c.company_id = companyId

// STEP 1.4: Organize qualifiers by rule_id into a lookup map
//   → Each rule gets a list of inclusion values and exclusion values
qualifiersByRule = new Map()
FOR EACH qualifier IN allQualifiers:
    IF qualifier.qualifier_type == "exclusion" OR qualifier.operator == "not_in":
        qualifiersByRule[qualifier.rule_id].exclusions.add(qualifier.qualifier_value)
    ELSE:
        qualifiersByRule[qualifier.rule_id].inclusions.add(qualifier.qualifier_value)

// Example map result:
// Rule A (Potatoes Rule): { inclusions: ["Potatoes", "Onions"], exclusions: [] }
// Rule B (General Rule):  { inclusions: ["General"],           exclusions: ["Potatoes","Onions"] }
// Rule C (Volume Rebate): { inclusions: [],                    exclusions: [] }  ← no qualifiers

// STEP 1.5: Group rules by contractId for efficient iteration
contractRulesMap = new Map()
FOR EACH rule IN allRules:
    contractRulesMap[rule.contractId].add(rule)

// Example map result:
// CNT-011: [Rule A (Potatoes), Rule B (General)]
// CNT-016: [Rule C (Volume Rebate)]
// CNT-012: [Rule D (...)]

Phase 2: The Main Matching Loop (Sale × Contract × Rule)

This is the core algorithm. It's a triple-nested loop: for each sale, try every contract, and within each contract try every rule. Track the best match with highest specificity score.

// ─── PHASE 2: MAIN MATCHING LOOP ─────────────────────────────────

allJunctionRecords = []  // ALL contract matches (for junction table)
primaryMatches     = []  // best match per sale (for matched_contract_id)
unmatchedIds       = []  // sale IDs that didn't match anything

// ┌─────────────────────────────────────────────────────────────────┐
// │  OUTER LOOP: Iterate through every unmatched sale              │
// │  (613 sales → 613 iterations)                                 │
// └─────────────────────────────────────────────────────────────────┘
FOR EACH sale IN unmatchedSales:

    saleMatches     = []   // collect ALL matches for this sale
    bestMatch       = null
    bestSpecificity = -1

    // Prepare lowercase versions of sale fields for case-insensitive matching
    saleProductLower  = sale.product_name.toLowerCase()   // "russet potatoes 50lb"
    saleCategoryLower = sale.category.toLowerCase()       // "potatoes"

    // ┌─────────────────────────────────────────────────────────────┐
    // │  MIDDLE LOOP: Try every contract for this company          │
    // │  (e.g., 3 contracts: CNT-011, CNT-012, CNT-016)           │
    // └─────────────────────────────────────────────────────────────┘
    FOR EACH (contractId, contractRules) IN contractRulesMap:

        // ── CHECK 1: Date Range Gate ──────────────────────────────
        // Does the sale's transaction_date fall within
        // the contract's effective_start and effective_end?
        // If NOT → skip this entire contract (all its rules)
        saleDate = sale.transaction_date
        IF contract.effective_start AND saleDate < contract.effective_start:
            CONTINUE // sale is before contract started → skip contract
        IF contract.effective_end AND saleDate > contract.effective_end:
            CONTINUE // sale is after contract ended → skip contract

        // ┌─────────────────────────────────────────────────────────┐
        // │  INNER LOOP: Try every rule in this contract            │
        // │  (e.g., CNT-011 has 2 rules: Potatoes Rule, General)   │
        // └─────────────────────────────────────────────────────────┘
        FOR EACH rule IN contractRules:

            // Call the rule-matching function (see Phase 2B below)
            result = saleMatchesRule(sale, rule, qualifiersByRule[rule.id])

            IF result.matches:
                // ── Collect EVERY match (for junction table) ──────
                saleMatches.add({
                    saleId:          sale.id,
                    contractId:      rule.contractId,
                    ruleId:          rule.id,
                    matchType:       result.reason,
                    specificityScore: result.specificity
                })

                // ── Track the best match (for matched_contract_id) ──
                IF result.specificity > bestSpecificity:
                    bestSpecificity = result.specificity
                    bestMatch = {
                        saleId:         sale.id,
                        contractId:     rule.contractId,
                        contractNumber: rule.contractNumber,
                        matchedRule:    rule.ruleName,
                        reason:         result.reason
                    }

    // ── After trying ALL contracts and ALL rules for this sale ───
    IF saleMatches.length > 0:
        // Mark the best match as is_primary = true
        FOR EACH match IN saleMatches:
            match.isPrimary = (match.specificityScore == bestSpecificity)
        allJunctionRecords.addAll(saleMatches) // ALL matches → junction table
        primaryMatches.add(bestMatch)              // best match → matched_contract_id
    ELSE:
        unmatchedIds.add(sale.id)                    // no contract matched this sale

Phase 2B: Inside saleMatchesRule() — The Rule Matching Decision Tree

For each (sale, rule) pair, this function decides: does this sale match this rule? And if yes, how specific is the match? It checks two layers in order: Qualifiers first, then Product Categories.

// ─── FUNCTION: saleMatchesRule(sale, rule, qualifiers) ───────────
// Returns: { matches: boolean, specificity: number, reason: string }

FUNCTION saleMatchesRule(sale, rule, qualifiers):

    saleProductLower  = sale.product_name.toLowerCase()   // "russet potatoes 50lb"
    saleCategoryLower = sale.category.toLowerCase()       // "potatoes"

    // ═══════════════════════════════════════════════════════════════
    // LAYER 1: CHECK QUALIFIERS (contract_qualifiers table)
    // This layer takes priority. If qualifiers exist for this rule,
    // they control the match decision completely.
    // ═══════════════════════════════════════════════════════════════

    IF qualifiers exist AND (has inclusions OR has exclusions):

        // ── STEP A: Check exclusions FIRST (blockers) ────────────
        // If any exclusion value matches the sale → BLOCKED, no match
        //
        // Matching is bidirectional "contains":
        //   "russet potatoes 50lb".includes("potatoes") → true (BLOCKED)
        //   "potatoes".includes("russet potatoes 50lb") → false
        //   Category "potatoes".includes("potatoes")    → true (BLOCKED)

        FOR EACH exclusionValue IN qualifiers.exclusions:
            exLower = exclusionValue.toLowerCase()

            // Check product_name (bidirectional contains)
            IF saleProductLower.includes(exLower) OR exLower.includes(saleProductLower):
                RETURN { matches: false, specificity: 0 }  // BLOCKED!

            // Check category (one-way contains)
            IF saleCategoryLower.includes(exLower):
                RETURN { matches: false, specificity: 0 }  // BLOCKED!

        // ── STEP B: Check inclusions (if any) ─────────────────────
        IF qualifiers.inclusions.length > 0:

            // Special case: inclusions = ["General"] → treat as catch-all
            // Skip inclusion matching, fall through to product_categories
            IF inclusions == ["General"]:
                // fall through to Layer 2 (product categories)

            ELSE:
                // Try to match each inclusion value against the sale
                // Same bidirectional "contains" logic as exclusions
                FOR EACH inclusionValue IN qualifiers.inclusions:
                    incLower = inclusionValue.toLowerCase()

                    IF saleProductLower.includes(incLower) OR incLower.includes(saleProductLower):
                        RETURN { matches: true,
                                 specificity: 1000 / qualifiers.inclusions.length,
                                 reason: "qualifier match" }

                    IF saleCategoryLower.includes(incLower):
                        RETURN { matches: true,
                                 specificity: 1000 / qualifiers.inclusions.length,
                                 reason: "qualifier match" }

                // No inclusion value matched → REJECTED
                RETURN { matches: false, specificity: 0 }

    // ═══════════════════════════════════════════════════════════════
    // LAYER 2: CHECK PRODUCT CATEGORIES (contract_rules.product_categories[])
    // Only reached if: no qualifiers exist, OR qualifiers were ["General"]
    // ═══════════════════════════════════════════════════════════════

    categories = rule.product_categories   // e.g. ["General","!Potatoes","!Onions"]

    IF categories is empty:
        RETURN { matches: false, specificity: 0 }  // no categories = no match possible

    // Split categories into inclusion list and exclusion list
    // Prefix "!" means exclusion (e.g., "!Potatoes" → exclude Potatoes)
    inclusionCats = []
    exclusionCats = []
    FOR EACH cat IN categories:
        IF cat starts with "!":
            exclusionCats.add( cat without "!" prefix )   // "!Potatoes" → "potatoes"
        ELSE:
            inclusionCats.add( cat )                      // "General" → "general"

    // ── STEP C: Check category exclusions ─────────────────────
    FOR EACH exCat IN exclusionCats:
        IF saleProductLower.includes(exCat) OR exCat.includes(saleProductLower):
            RETURN { matches: false, specificity: 0 }  // product excluded!
        IF saleCategoryLower.includes(exCat):
            RETURN { matches: false, specificity: 0 }  // category excluded!

    // ── STEP D: Check category inclusions ─────────────────────
    IF inclusionCats.length > 0:

        // Special case: categories = ["General"] → universal catch-all
        IF inclusionCats == ["General"]:
            RETURN { matches: true, specificity: 1, reason: "general rule" }

        // Try to match each category against the sale
        FOR EACH catName IN inclusionCats:
            IF catName == "general": SKIP

            // Bidirectional contains:
            //   sale "soundpro wireless headphones - black"
            //   cat  "soundpro wireless headphones - black"
            //   → product.includes(cat) = true ✓
            IF saleProductLower.includes(catName) OR catName.includes(saleProductLower):
                RETURN { matches: true,
                         specificity: 1000 / inclusionCats.length,
                         reason: "product/category match" }

            IF saleCategoryLower.includes(catName) OR catName.includes(saleCategoryLower):
                RETURN { matches: true,
                         specificity: 1000 / inclusionCats.length,
                         reason: "product/category match" }

    // Nothing matched at all
    RETURN { matches: false, specificity: 0 }

Specificity Scoring — Why It Matters

A single sale can match rules in multiple contracts. The specificity score ensures the most specific contract wins. The score formula is simple: 1000 / number_of_values. Fewer values = more specific = higher score.

The outer loop collects every contract+rule match for each sale into a saleMatches list. All of these are written to the sale_contract_matches junction table. If CNT-011 matches with score 500 and CNT-012 also matches with score 1 (General rule), both are recorded. CNT-011 is marked is_primary = true and written to matched_contract_id. One Sale → Many Contracts (Junction Table Architecture)

shared/schema.ts — saleContractMatches  |  server/services/salesContractMatcher.ts  |  server/storage.ts — getSalesForContractViaJunction()

Architectural Change: From 1:1 to 1:N Matching

Previously, each sale could only belong to one contract via the sales_data.matched_contract_id column. This caused a limitation: if a sale legitimately owed fees to multiple contracts (e.g., a vendor rebate AND a marketing fee AND a volume incentive), only the "best" contract would calculate fees.

The new junction table (sale_contract_matches) stores every contract+rule match for each sale, enabling one sale to generate fees across multiple contracts independently.

5 Match Each Sale to a Rule (Multi-Dimensional Qualifier Enforcement)

server/services/calculationService.ts — ruleMatchesSaleByQualifiers()

For each sale, the system walks through rules in priority order and applies multi-dimensional filtering. A sale must pass ALL qualifier dimensions to match a rule.

Complete Matching Logic — All Qualifier Dimensions (in order)

  1. Product exclusions (contract_qualifiers table): If the sale's product_name or category matches any exclusion value → skip this rule
  2. Product inclusions (contract_qualifiers table): If the rule has specific inclusions (not just "General"), the sale must match at least one inclusion value
  3. Product categories (contract_rules.product_categories): Fallback product matching with ! prefix for exclusions. ["General"] = wildcard.
  4. Territory filter (contract_rules.territories): If the rule has a non-empty territories array (and not ["all"]), the sale's territory must match one of the listed values (case-insensitive)
  5. Channel filter (contract_rules.channel): If the rule has a non-empty channel string, the sale's channel must match (case-insensitive)
  6. Customer segment filter (contract_rules.customer_segments): Reserved for future use — sales_data does not yet have a segment column
  7. Date range filter (contract_rules.effective_date / expiry_date): Sale's transaction_date must fall within the rule's date range (if set)
  8. First match wins: Once a sale matches a rule, processing stops for that sale (break)

Formula Resolution Priority

The engine tries to build an executable formula in this order:

  1. formula_definition with formula node: Use the stored JSON expression tree directly via the Universal Formula Evaluator
  2. formula_definition with tableData + fieldMappings: Build a formula on-the-fly from extracted table data
  3. formula_definition with baseRate: Build a simple formula — for fixed_fee/per_unit types: quantity × fixedAmount; for percentage types: grossAmount × baseRate
  4. formula_definition with fixedAmount: Build quantity × fixedAmount, optionally gated by a threshold condition
  5. volume_tiers array: Build a tiered lookup formula from the tiers
  6. Final fallback — base_rate column: Uses base_metric to determine the correct sale column:
    • If base_metric is units, quantity, or unit — OR rule_type is fixed_fee, per_unit, or per-unit: formula = quantity × base_rate
    • Otherwise (percentage, gross_sales, net_sales, etc.): formula = grossAmount × base_rate

Post-Calculation Checks

  • If minimum_guarantee is set and the result is less → use the minimum guarantee instead
  • If minimum_price is set and the result is less → use the minimum price instead

Retroactive vs Non-Retroactive

Retroactive Mode (default)

When cumulative total reaches a higher tier, the new tier rate is applied to ALL prior volume.

Example: 10,000 units at Tier 1 (2%), then hit 15,000 → Tier 2 (4%) applies to all 15,000 units.

// Retroactive: apply matched tier rate to ALL cumulative volume
milestoneAmount = isPercentage ? totalRevenue × tierRate : tierRate × matchingSales.length;

Non-Retroactive (Marginal) Mode

Each sale is processed individually at the tier rate for its cumulative position.

// Non-retroactive: each sale processed at running cumulative tier
for (sale of matchingSales) {
  cumulativeTotal += saleValue;
  matchedTier = findTier(cumulativeTotal, sortedTiers);
  saleAmount = isPercentage ? saleValue × tierRate : tierRate;
}

Key Concept: Pre-Aggregated vs Per-Sale

Per-sale tiered (tiered): Each sale is matched to a tier independently. Sale A with 50 units might hit Tier 1; Sale B with 200 units might hit Tier 2. They don't affect each other.

Pre-aggregated tiered (rebate_tiered, milestone_tiered): ALL matching sales are summed up first into a total, THEN the total determines which tier applies, THEN the tier rate is applied back to the total (or to each sale). Sales affect each other's tier because they're summed before tier lookup.

Type 1: Per-Sale Tiered (tiered / tiered_pricing)

The simplest tiered type. Each sale is independently checked against the tier brackets.

Tier Data Structure (volume_tiers)

// Stored in contract_rules.volume_tiers (jsonb)
[
  { "min": 0,      "max": 100,   "rate": 0.50 },   // 0-100 units: $0.50/unit
  { "min": 101,    "max": 500,   "rate": 0.40 },   // 101-500 units: $0.40/unit
  { "min": 501,    "max": null,  "rate": 0.30 }    // 501+ units: $0.30/unit
]

Algorithm: findMatchingTier()

function findMatchingTier(value, tiers) {
  // Sort by min DESCENDING (highest threshold first)
  const sorted = tiers.sort((a, b) => b.min - a.min);
  // Walk from highest to lowest, first match wins
  for (const tier of sorted) {
    if (value >= tier.min) return tier;  // ← found the tier!
  }
  // Fallback: if nothing matched, use lowest tier
  return sorted[sorted.length - 1];
}

Important: "Whole Tier" Not "Marginal"

Per-sale tiered uses whole-tier logic: the entire sale quantity is multiplied by the matched tier's rate. It does NOT use marginal/graduated logic (where the first 100 units get one rate and units 101-500 get a different rate). For Sale B (300 units), ALL 300 units get $0.40 — not 100 @ $0.50 + 200 @ $0.40.

Type 2: Quarterly Rebate Tiered (rebate_tiered)

The most complex tiered type. Sales are aggregated by quarter before tier determination.

Filter sales by rule scope → Aggregate by quarter + product → Determine tier basis (units vs dollars) → Find tier per quarter → Apply tier rate to each product's total

Step-by-Step Processing

Step 1: Filter Sales by Rule Scope

Only sales matching the rule's product_categories and territories are included. Sales outside the rule's scope are completely ignored.

// Rule scope check (for each sale)
saleMatchesRuleScope(sale):
  // Check product match (fuzzy: includes, contains, or exact)
  IF rule.productCategories.length > 0:
    IF NOT saleCat matches any ruleCategory: RETURN false
  // Check territory match
  IF rule.territories.length > 0:
    IF NOT saleTerritory matches any ruleTerritory: RETURN false
  RETURN true
Step 2: Aggregate by Quarter + Product

Matching sales are grouped into quarterly buckets. Both dollar totals and unit totals are tracked per quarter.

// Quarter determination
quarter = sale.customFields.quarter OR derive(transactionDate)
  // Jan-Mar = Q1, Apr-Jun = Q2, Jul-Sep = Q3, Oct-Dec = Q4

// Two parallel aggregation maps:
quarterlyDollarTotals: { "2026-Q1": $145,000, "2026-Q2": $210,000 }
quarterlyUnitTotals:   { "2026-Q1": 8,500,    "2026-Q2": 12,300 }

// Also tracks per-product within each quarter (for per-product rebate lines):
quarterProductTotals: {
  "2026-Q1|Russet Potatoes": { netPurchases: $85,000, totalUnits: 5,000 },
  "2026-Q1|Yellow Onions":   { netPurchases: $60,000, totalUnits: 3,500 },
  "2026-Q2|Russet Potatoes": { netPurchases: $120,000, totalUnits: 7,000 },
  ...
}
Step 3: Determine Tier Basis (Units vs Dollars)

The engine auto-detects whether the tiers are unit-based or dollar-based by looking at the tier thresholds:

// Heuristic: if the highest tier min is under 500,000 → assume units
maxTierMin = Math.max(...tiers.map(t => t.min))
isUnitBased = maxTierMin > 0 AND maxTierMin < 500000
tierBasis = isUnitBased ? "units" : "dollars"
Step 4: Find Tier Per Quarter

Using the quarterly total (units or dollars), find the matching tier for each quarter:

// Rate normalization: supports both decimal (0.04) and percentage (4) storage
applicableRate = tier.rate < 1 ? tier.rate : tier.rate / 100
// 0.04 stays as 0.04 (4%)
// 4 becomes 4/100 = 0.04 (4%)

// Sort tiers ascending, walk from highest to lowest
FOR i = sortedTiers.length - 1 DOWN TO 0:
  IF quarterTotal >= sortedTiers[i].min:
    matchedTier = sortedTiers[i]
    BREAK

The matched tier rate is applied to each product's net purchases within that quarter:

productRebate = productData.netPurchases × applicableRate

The is_primary column answers one question: "Of all the contracts this sale matched, which one was the most specific match?"

Purpose 1: Backward Compatibility (matched_contract_id)

Before the junction table existed, each sale could only link to one contract via the sales_data.matched_contract_id column. The primary match is the one written to that column, so older code that reads matched_contract_id still works correctly.

Purpose 2: Reporting & Analytics

When showing "which contract is this sale's main match?" in the UI (e.g., Data Ingestion grid), the system can query WHERE is_primary = true to get a single answer per sale — even though the sale participates in fee calculations for multiple contracts.

What is_primary Does NOT Do

Critical: is_primary does NOT affect fee calculations

When the engine calculates fees for Contract B, it queries: SELECT sale_id FROM sale_contract_matches WHERE contract_id = Contract_B. There is no filter on is_primary. Every sale matched to Contract B participates in its calculation — regardless of whether Contract B was the primary, secondary, or tertiary match for that sale.

How Each Contract Uses This Sale

  • Contract A queries WHERE contract_id = Contract_A → finds abc123 → calculates 3.5% on $2,220 = $77.70
  • Contract B queries WHERE contract_id = Contract_B → finds abc123 → calculates 2% on $2,220 = $44.40
  • Contract C queries WHERE contract_id = Contract_C → finds abc123 → calculates 1% on $2,220 = $22.20

The same sale generates fees across all three contracts. Contract A was the "primary" (score 1000 > 1), but is_primary is irrelevant to the fee calculation — it's only used for matched_contract_id and display purposes.

Duplicate Prevention on Upload

server/routes.ts — POST /api/sales/upload

Problem: Orphaned Sales Data on Re-Upload

Previously, uploading a CSV file, deleting the dataset record, and uploading the same file again would create duplicate sales rows. The dataset delete only removed the uploaded_datasets metadata record — it did not delete the underlying sales_data rows. This caused calculations to produce doubled results (e.g., $8,857 instead of the correct $4,428.55).

Solution: Auto-Clear Before Insert

The upload endpoint now automatically deletes existing sales data for the same scope before inserting new rows. This ensures that re-uploading a file always results in exactly one copy of the data.

Upload ModePre-Insert CleanupSQL Executed
Company-Wide Delete all sales_data for the user's active company DELETE FROM sales_data WHERE company_id = $1
Contract-Specific Delete all sales_data matched to that contract DELETE FROM sales_data WHERE matched_contract_id = $1

Idempotent Uploads

Uploading the same 116-row CSV file 5 times in a row will always produce exactly 116 rows in sales_data — never 232, 348, or more. Each upload is a complete replace operation for that scope, not an append.

Dataset Delete Cascade

server/routes.ts — DELETE /api/uploaded-datasets/:id

When a user deletes an uploaded dataset record from the Data Ingestion tab, the system now cascades the delete to remove the associated sales data rows.

Cascade Logic

Dataset TypeCascade ActionSQL Executed
Company-Wide (company_wide = true) Delete all sales for that company DELETE FROM sales_data WHERE company_id = $1
Contract-Specific (contract_id set) Delete sales matched to that contract DELETE FROM sales_data WHERE matched_contract_id = $1

Lookup Before Delete

The endpoint first queries SELECT company_id, contract_id, company_wide FROM uploaded_datasets WHERE id = $1 to determine the scope, then deletes the associated sales data, and finally deletes the metadata row from uploaded_datasets.

Table Relationship

uploaded_datasets → sales_data (via company_id or contract_id)

Note: There is no direct foreign key between uploaded_datasets and sales_data. The linkage is implicit through company_id (for company-wide uploads) or matched_contract_id (for contract-specific uploads).
