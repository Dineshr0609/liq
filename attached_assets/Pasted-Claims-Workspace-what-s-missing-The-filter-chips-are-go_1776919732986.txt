Claims Workspace — what's missing
The filter chips are good but there's no KPI strip showing Finance the headline numbers before they drill into the queue — how many urgent, how many agent-handling, total open exposure, and the agent's current activity status. Opening this screen right now gives you no situational awareness until you start clicking. The agent panel says "The agent will narrate decisions here" which is good as an empty state but the agent bar across the top showing live activity (what the agent is processing right now, auto-resolve rate, items handled today) is missing — that's the signal that the system is working even when Finance isn't.
The Simulate button is interesting — what does it do? If it simulates a claim calculation against contract rules that's genuinely useful. Make sure it's clear in the UI what simulation means — Finance might confuse it with running a real calculation.
The Queue/Period/Contract view switcher is correct and matches what we designed. Good.
Missing: there's no indication of inbound vs outbound direction in the empty queue label. When Finance first opens this, they should understand the two-directional nature of claims without having to filter first.

Invoices & Memos — what's missing
The tab structure (AP Invoices / AR Invoices / Credit Memos / Debit Memos) is exactly right. The filter chips (Draft / Awaiting Oracle / Posted) are correct.
Two things missing here. First, there's no KPI strip — Finance opening Invoices & Memos has no idea of their current position. How many AP invoices are open? What's the total payable? What's overdue? That summary matters at a glance. Second, there's no Oracle sync status indicator anywhere visible — Finance needs to know whether the Oracle connection is live and when it last synced. If there's a sync failure, they need to know before they assume a document is posted when it isn't.
Also — the "+ New" button in the AP Invoices queue: when would Finance manually create an AP invoice? Most invoices should be auto-generated by the agent from approved claims. If manual creation is allowed, it needs a prominent warning that manually created invoices bypass the claim validation chain and won't have full traceability. Or consider renaming it "Override" to signal that it's an exceptional action.

Deductions — what's missing
The description "Review customer short-pays and chargebacks; match, dispute, write off, or recover" is excellent — most precise description of the three screens. Keep that exactly.
The filter chips are good: Open, Needs Review, Disputed, Matched, Written off, Recovered. These cover the full lifecycle.
What's missing: the Expected vs Unexpected split we discussed. Right now the queue shows all deductions as one undifferentiated list. The most important cognitive distinction for Finance is "which of these did we know were coming (expected, auto-clearable) vs which surprised us (unexpected, need decomposition)." Without that split, Finance has to manually inspect every item to understand its urgency. Add a view switcher — Unexpected / Expected / All — or at minimum a badge on each queue item indicating whether it's expected or not.
Also missing: no agent activity indicator. The agent should be auto-matching expected deductions in the background. Finance should see this happening — "Agent auto-cleared 34 expected deductions today" — so they trust the queue showing only the 11 that actually need their attention.
The scissors icon for empty state is good — distinctive and thematically appropriate.
There's no write-off threshold policy visible anywhere. In practice, Finance has a materiality threshold — deductions under $500 might get auto-written off rather than disputed. That policy should be configurable and visible on this screen so Finance knows the agent is applying it.
