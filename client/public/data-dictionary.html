1. Dashboard KPIs — Contract Portfolio Overview

READ
Use Case: The Dashboard screen loads aggregate KPIs showing total contracts, pending approvals, and total license fee revenue for the active company. This query powers the hero stat cards.
Screen: Dashboard (/) • Tables: contracts, contract_royalty_calculations
SELECT COUNT(c.id) AS total_contracts, COUNT(CASE WHEN c.status = 'analyzed' THEN 1 END) AS analyzed, COUNT(CASE WHEN c.approval_state = 'pending' THEN 1 END) AS pending_approval, COALESCE(SUM(calc.total_royalty), 0) AS total_license_fees FROM contracts c LEFT JOIN contract_royalty_calculations calc ON calc.contract_id = c.id AND calc.status = 'approved' WHERE c.company_id = '40405694-...' -- Active company context ;

6. ERP Field Mapping — Source to Target Translation

READ
Use Case: The Master Data Mapping screen shows how ERP fields map to LicenseIQ fields. This query resolves the full chain: ERP System → Entity → Field → Mapping → LicenseIQ Entity → Field.
Screen: Master Data Mapping (/master-data-mapping) • Tables: master_data_mappings, erp_systems, erp_entities, erp_fields, licenseiq_entities, licenseiq_fields
SELECT es.name AS erp_system, ee.name AS erp_entity, ef.field_name AS erp_field, ef.data_type AS erp_type, ' → ', le.name AS liq_entity, lf.field_name AS liq_field, lf.data_type AS liq_type, mdm.ai_confidence FROM master_data_mappings mdm JOIN erp_systems es ON es.id = mdm.erp_system_id JOIN licenseiq_entities le ON le.id = mdm.licenseiq_entity_id, LATERAL jsonb_array_elements(mdm.mapping_results->'fieldMappings') AS fm LEFT JOIN erp_entities ee ON ee.system_id = es.id LEFT JOIN erp_fields ef ON ef.entity_id = ee.id AND ef.field_name = fm->>'sourceField' LEFT JOIN licenseiq_fields lf ON lf.entity_id = le.id AND lf.field_name = fm->>'targetField' WHERE mdm.status = 'approved';

7. Multi-Location Access — User Context Resolution

READ
Use Case: When a user switches company/business unit in the top nav, the system resolves their active context to filter all data. This query shows how the context chain works: user → active context → org role → company/BU/location.
Screen: All screens (context switcher) • Tables: users, user_active_context, user_organization_roles, companies, business_units, locations
SELECT u.username, u.role, uor.role AS org_role, co.company_name, bu.org_name AS business_unit, loc.loc_name AS location FROM users u JOIN user_active_context uac ON uac.user_id = u.id JOIN user_organization_roles uor ON uor.id = uac.active_org_role_id JOIN companies co ON co.company_id = uor.company_id LEFT JOIN business_units bu ON bu.org_id = uor.business_unit_id LEFT JOIN locations loc ON loc.loc_id = uor.location_id WHERE u.id = 'user-789';

8. liQ AI — RAG Vector Similarity Search

READ
Use Case: When a user asks a question in liQ AI, the system converts the question to an embedding vector and finds the most similar contract sections using pgvector cosine distance. The top results are sent as context to the AI model.
Screen: liQ AI (/contract-qna) • Tables: contract_embeddings, contracts
SELECT ce.source_text, ce.metadata, c.display_name, c.contract_number, 1 - (ce.embedding <=> '[0.023, -0.041, ...]'::vector) AS similarity FROM contract_embeddings ce JOIN contracts c ON c.id = ce.contract_id WHERE c.company_id = '40405694-...' ORDER BY ce.embedding <=> '[0.023, -0.041, ...]'::vector LIMIT 5; -- Top 5 most relevant chunks
Example Result:

royalty_rules.fieldMappings JSONB Maps extracted table column headers → calculation engine field names

When AI extracts a table from a contract PDF, the column headers are arbitrary (e.g., "Quarterly Net Purchases", "Royalty %"). The fieldMappings column stores which extracted column maps to which internal calculation field. This mapping is set by the user in the DynamicTierTable component via dropdown selectors.

JSONB KeyExpected ValueMaps ToDescription
volumeFieldstringVolume / quantity columne.g., "Quarterly Net Purchases" → used as the lookup dimension in tier matching
rateFieldstringRate / percentage columne.g., "Royalty %" → the rate value applied to sales amount
minFieldstringTier minimum columne.g., "From" → lower bound of a volume tier

royalty_rules.formulaDefinition JSONB Contains raw table data + executable expression tree for the calculation engine

This is the most complex JSONB column. It stores two things: (1) the raw tableData extracted from the contract PDF, and (2) an expression tree that the Universal Formula Evaluator uses to calculate license fees. When fieldMappings are saved, the backend calls buildFormulaDefinition() to auto-generate the expression tree from the table data + mappings.

JSONB PathTypeDescription
tableDataobjectRaw extracted table with headers[] and rows[][] from the PDF
  tableData.headersstring[]Column headers, e.g., ["Quarterly Net Purchases", "From ($)", "To ($)", "Royalty %"]
  tableData.rowsstring[][]Each row is an array of cell values, e.g., [["Tier 1", "0", "5000", "5%"]]
expressionobjectExecutable expression tree (nested node structure)
  expression.typestringNode type: "multiply", "tier", "lookup", "field", "constant"
  expression.childrenarrayChild nodes forming the calculation tree
  expression.tiersarrayFor tier nodes: [{min, max, rate}] derived from tableData + fieldMappings
sourceTablestringWhich PDF table this came from (section reference)

Rules Field Mapping → Calculation Data Flow 
1. AI Extraction (contract upload)
   Contract PDF → AI extracts tables → royalty_rules.formula_definition.tableData
                                           → royalty_rules.base_rate, product_categories, territories

2. User Maps Fields (Rules screen → DynamicTierTable dropdowns)
   User selects: "Quarterly Net Purchases" → volumeField
                 "Royalty %"               → rateField
   Saved to → royalty_rules.field_mappings (JSONB)

3. Expression Tree Built (backend: buildFormulaDefinition)
   tableData + fieldMappings → formulaDefinition.expression (executable tree)
   Also updates → royalty_rules.volume_tiers (simplified tier array)

4. Calculation Engine (License Fee Calculator screen)
   sales_data row → matched to royalty_rules via contract + territory + product
                  → evaluateFormula(expression, salesRow) → calculated fee
                  → calculation_line_items (audit trail per transaction)
